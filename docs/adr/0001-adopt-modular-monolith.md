# ADR-001 モジュラーモノリスを採用する

## ステータス
採択

## コンテキスト
個人開発で銀行 API を叩く実装を構築する。将来的に本番 BaaS 移行や複数決済代行業者の追加も視野に入れたいが、現時点ではマイクロサービスのインフラ運用コストは過大。一方で全モジュールを単一 package にまとめると、機能追加時の変更影響が広がりテスト容易性が損なわれる。

## 検討した選択肢
- 案 A 単一 package のモノリス
  - 利点 立ち上がりが最速、ファイル数最小
  - 欠点 銀行 API 追加・消込の追加で全 package に変更影響、ドメイン境界が曖昧
- 案 B マイクロサービス
  - 利点 完全な独立、技術選択の自由度
  - 欠点 個人開発でインフラ運用が過大、分散トランザクションが必要
- 案 C モジュラーモノリス ( 4 モジュール、共通 DB、共通 Outbox )
  - 利点 単一デプロイの簡便さを残しつつ、ドメイン境界をコードで強制可能
  - 欠点 モジュール公開 API の設計が必要、規律のためのレビューが必要

## 決定
案 C を採択する。Account / Transfer / Reconciliation / Notification の 4 モジュールに分け、各モジュールは port.go に定義した interface でのみ他モジュールから参照される構造にする。

## 影響
- internal/modules/* 配下に各モジュールを配置
- 横断関心事は internal/platform/* に集約
- モジュール間の連携は internal/scenario/ で薄く調整
- 共通 Outbox ( internal/platform/outbox ) が中央のイベントバス的役割を担う
